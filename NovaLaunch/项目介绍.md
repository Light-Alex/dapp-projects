# NovaLaunch 项目介绍

---

## 1. 项目概述

### 1.1 项目定位

**NovaLaunch** 是一个基于区块链技术的去中心化代币发行平台（Launchpad），专注于为优质区块链项目提供代币发行服务，同时为投资者提供参与早期项目投资的机会。

### 1.2 核心价值

平台的核心创新在于**质押权重分配机制**：投资者通过质押平台代币（Nova）获得参与 IDO（Initial DEX Offering）的购买权重，质押数量越多，获得的项目代币分配额度越高。这种机制既保证了项目的真实用户基础，又防止了短期投机行为。

### 1.3 技术栈总览

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| 智能合约层 | Solidity 0.6.12, Hardhat, OpenZeppelin | 业务逻辑核心，部署于区块链网络 |
| 前端层 | Next.js, React, TypeScript, Ant Design, Web3.js | 用户交互界面 |
| 后端层 | Java, Spring Boot, Spring Cloud, Web3j, MySQL | 业务数据管理和 API 服务 |
| 数据索引 | The Graph | 区块链事件索引和查询 |

## 2. 业务介绍

### 2.1 什么是 IDO

**IDO（Initial DEX Offering）** 即去中心化交易所公开发行，是一种代币发行方式。与传统 ICO 不同，IDO 在去中心化交易所上进行，具有以下特点：

- **无需中心化中介**：通过智能合约自动执行
- **即时流动性**：发行后即可在 DEX 交易
- **公平分配**：通过质押机制确保真实用户参与

### 2.3 完整业务流程

1. **项目申请阶段**：项目方提交项目信息，平台管理员进行审核
2. **准备阶段**：审核通过后，设置销售参数（价格、数量、时间等）
3. **用户质押阶段**：投资者质押 Nova 代币获得购买权重
4. **注册阶段**：用户在指定时间窗口内注册参与 IDO
5. **销售阶段**：注册用户购买项目代币
6. **释放阶段**：按线性释放计划逐步解锁购买的代币

**合约实现**：`contracts/Admin.sol`

```solidity

modifier onlyAdmin {
    require(isAdmin[msg.sender], "Only admin can call.");
    _;

```

#### 4.2.2 项目方/销售所有者 (SaleOwner)

**定位**：代币发行方，通过平台筹集资金的项目团队。

**权限矩阵**：

| 权限 | 对应函数 | 说明 |
|------|----------|------|
| 存入销售代币 | depositTokens | 将待销售代币转入合约 |
| 提取收益 | withdrawEarningsAndLeftover | 销售结束后提取 ETH 和剩余代币 |

**合约实现**：`contracts/sales/NovaSale.sol`

```solidity
function depositTokens() external onlySaleOwner {
    require(!sale.tokensDeposited, "Deposit can be done only once");
    sale.tokensDeposited = true;
    sale.token.safeTransferFrom(msg.sender, address(this), sale.amountOfTokensToSell);
}
```

**限制**：
- 只能在销售结束后提取收益
- 每个销售合约只能提取一次

#### 4.2.3 质押者 (Staker)

**定位**：平台代币（Nova）持有者，通过质押获得参与 IDO 的权重。

**权重层级系统**：

| 层级 | 质押数量 (Nova) | 权重值 |
|------|----------------|--------|
| Tier One | 1 - 5,000 | 1 |
| Tier Two | 5,001 - 25,000 | 2 |
| Tier Three | 25,001 - 50,000 | 3 |
| Tier Four | 50,001 - 100,000 | 4 |
| Tier Five | > 100,000 | 5 |

**权限矩阵**：

| 权限 | 对应函数 | 说明 |
|------|----------|------|
| 质押代币 | deposit | 将 Nova 存入质押池 |
| 提取质押 | withdraw | 取出质押的代币（销售结束后） |
| 领取奖励 | withdraw（自动计算） | 领取质押产生的奖励 |
| 复投奖励 | compound | 将奖励自动复投到质押中 |
| 紧急提取 | emergencyWithdraw | 放弃奖励，只取本金 |

**合约实现**：`contracts/AllocationStaking.sol`

#### 4.2.4 购买者/参与者 (Participant)

**定位**：参与 IDO 的投资者，通过质押获得购买资格。

**权限矩阵**：

| 权限 | 对应函数 | 说明 |
|------|----------|------|
| 注册参与 | registerForSale | 在注册窗口内报名参与 IDO |
| 购买代币 | participate | 使用 ETH 购买项目代币 |
| 提取代币 | withdrawTokens | 按释放计划提取购买的代币 |
| 批量提取 | withdrawMultiplePortions | 一次性提取多个已解锁部分 |

**限制**：
- 必须先质押 Nova 代币
- 必须在注册窗口内完成注册
- 注册后质押将被锁定直到销售结束
- 每个销售只能参与一次

### 5.2 Airdrop流程详细说明

Airdrop 合约用于向早期用户免费分发代币，作为社区激励和用户获取代币的手段。

#### 阶段 1：部署Airdrop合约（Admin）

```
1. Admin 部署 Airdrop 合约
   构造参数：
   - airdropToken: 空投代币合约地址（Nova）

2. Admin 向 Airdrop 合约转入空投代币
   - 转账数量：根据预计领取用户数量计算
   - 公式：所需代币 = 预计用户数 × TOKENS_PER_CLAIM
   - 例如：100个用户 × 100代币 = 10,000 代币
```

#### 阶段 2：用户领取空投（User）

用户只能领取一次空投。

```
1. User 调用 Airdrop.withdrawTokens() 领取空投
   无需参数，自动向 msg.sender 发放代币

2. 合约执行流程：
   a) 检查 msg.sender == tx.origin（防止合约调用）
   b) 检查 !wasClaimed[beneficiary]（防止重复领取）
   c) 标记用户已领取：wasClaimed[msg.sender] = true
   d) 转账 TOKENS_PER_CLAIM 数量的代币给用户
   e) 更新 totalTokensWithdrawn

3. 触发事件：TokensAirdropped(beneficiary)
```

#### Airdrop 合约核心机制

**固定数量空投：**

```solidity
uint256 public constant TOKENS_PER_CLAIM = 100 * 10**18;
```

每个地址可领取 100 个代币（可调整常量值）

**防止合约调用：**

```solidity
require(msg.sender == tx.origin, "Require that message sender is tx-origin.");
```

确保只有外部账户（EOA）可以领取，防止恶意合约批量薅羊毛

**防止重复领取：**

```solidity
mapping (address => bool) public wasClaimed;
require(!wasClaimed[beneficiary], "Already claimed!");
```

**关键数据结构：**

```solidity
// 状态变量
IERC20 public airdropToken;          // 空投代币合约
uint256 public totalTokensWithdrawn;  // 总提取数量
mapping (address => bool) public wasClaimed;  // 领取记录
```

#### 查询函数

| 函数 | 返回值 | 说明 |
|------|--------|------|
| wasClaimed(address) | bool | 查询指定地址是否已领取 |
| totalTokensWithdrawn() | uint256 | 查询已发放的代币总量 |
| airdropToken() | address | 查询空投代币合约地址 |

#### 使用场景

1. **早期用户奖励**：奖励平台早期支持者
2. **社区激励**：扩大社区影响力
3. **代币分发**：无需 gas 费的批量分发方式（用户自行领取）
4. **流量获取**：吸引新用户注册使用平台

#### 注意事项

- 空投合约不可升级，部署后参数不可修改
- 每个地址只能领取一次
- 需确保合约内有足够代币余额
- 建议配合白名单机制使用（当前版本为公开领取）


### 5.3 Farm流程详细说明

NovaLaunch 有两个质押挖矿合约：

| 合约名称 | 文件路径 | 主要用途 | 可升级性 |
|---------|----------|----------|----------|
| AllocationStaking | contracts/AllocationStaking.sol | 分配质押，用于获取 IDO 参与权重 | 可升级（代理模式） |
| FarmingNova | contracts/farming/FarmingNova.sol | 流动性挖矿，纯收益 farming | 不可升级 |

#### 阶段 1：部署Farm合约（Admin）

```
1. Admin 调用 deploy_farm.js 部署 FarmingNova 合约
   构造参数：
   - erc20: 奖励代币合约地址（Nova）
   - rewardPerSecond: 每秒奖励数量（如 1 ERC20/秒）
   - startTimestamp: 质押开始时间戳

2. Admin 调用 fund(_amount) 向 Farm 合约添加奖励代币
   - 延长 endTimestamp = endTimestamp + _amount / rewardPerSecond
   - 增加 totalRewards

3. Admin 调用 add(_allocPoint, _lpToken, _withUpdate) 添加 LP 池子
   - allocPoint: 分配点数（权重，如 100）
   - lpToken: LP 代币合约地址
   - withUpdate: 是否更新所有池子奖励
```

#### 阶段 2：用户质押（User）

```
1. User 调用 FarmingNova.deposit(pid, amount)
   参数：
   - pid: 池子 ID（从 0 开始）
   - amount: 质押的 LP 代币数量

2. 合约执行流程：
   a) 调用 updatePool(pid) 更新池子奖励
   b) 如果用户之前有质押，计算并发送待领取奖励
   c) 从用户转入 LP 代币到合约
   d) 更新用户质押数量 user.amount
   e) 重新计算 user.rewardDebt

3. 用户获得奖励代币的收益权
```

#### 阶段 3：领取奖励/取出质押（User）

```
方式一：取出并领取奖励
1. User 调用 FarmingNova.withdraw(pid, amount)
2. 合约执行：
   a) 更新池子奖励
   b) 计算用户待领取奖励：pending = user.amount × accERC20PerShare / 1e36 - user.rewardDebt
   c) 发送奖励给用户
   d) 返还用户指定的 LP 代币数量
   e) 更新用户信息和池子总质押

方式二：复投奖励
1. User 调用 AllocationStaking.compound(pid)
2. 合约执行：
   a) 更新池子奖励
   b) 计算待领取奖励
   c) 将奖励直接加入用户质押数量（不转账）
   d) 更新 rewardDebt
   e) 更新池子总质押

方式三：紧急取款（放弃奖励）
1. User 调用 FarmingNova.emergencyWithdraw(pid)
2. 合约执行：
   a) 只退还用户质押的 LP 代币
   b) 清零用户 amount 和 rewardDebt
   c) 不发放任何奖励
```

#### 阶段 4：管理操作（Owner）

```
1. 添加新池子：add(allocPoint, lpToken, withUpdate)
2. 调整池子权重：set(pid, allocPoint, withUpdate)
3. 追加奖励资金：fund(amount)
4. 批量更新池子：massUpdatePools()
```

#### Farm 合约核心算法

**待领取奖励计算公式：**

```
pending = (user.amount × pool.accERC20PerShare) ÷ 1e36 - user.rewardDebt
```

**accERC20PerShare 更新公式：**

```
nrOfSeconds = min(block.timestamp, endTimestamp) - pool.lastRewardTimestamp
erc20Reward = nrOfSeconds × rewardPerSecond × pool.allocPoint ÷ totalAllocPoint
pool.accERC20PerShare += erc20Reward × 1e36 ÷ pool.totalDeposits
```

**关键数据结构：**

```solidity
// 用户信息
struct UserInfo {
    uint256 amount;         // 用户质押的 LP 代币数量
    uint256 rewardDebt;     // 奖励债务（防止重复计算）
}

// 池子信息
struct PoolInfo {
    IERC20 lpToken;             // LP 代币合约地址
```

### 5.4 IDO流程详细说明

#### 阶段 1：项目准备（Admin + SaleOwner）

```
1. Admin 调用 SalesFactory.deploySale() 创建新销售合约
2. Admin 调用 setSaleParams() 设置：
   - token: 销售代币地址
   - saleOwner: 项目方地址
   - tokenPriceInETH: 代币价格
   - amountOfTokensToSell: 销售数量
   - saleEnd: 销售结束时间
   - tokensUnlockTime: 代币解锁时间
   - maxParticipation: 单用户最大购买额

3. Admin 调用 setVestingParams() 设置释放计划
4. Admin 调用 setRegistrationTime() 设置注册窗口
5. SaleOwner 调用 depositTokens() 存入待销售代币
```

#### 阶段 2：用户质押（User）

```
1. User 调用 AllocationStaking.deposit(pid, amount)
2. 合约更新用户质押数量
3. 计算并更新 rewardDebt
4. 用户获得对应层级的购买权重
```

#### 阶段 3：注册参与（User + Admin 签名）

```
1. 用户在注册窗口内向后端请求注册签名
2. 后端验证用户质押量和资格
3. 后端使用管理员私钥生成签名
4. User 调用 registerForSale(signature, pid)
5. 合约验证签名有效性
6. 锁定用户质押直到销售结束
```

#### 阶段 4：参与购买（User + Admin 签名）

```
1. Admin 调用 setSaleStart() 启动销售
2. 用户在销售期间内向后端请求购买签名
3. 后端计算用户最大可购买额度
4. 后端生成签名
5. User 调用 participate(signature, amount) 并发送 ETH
6. 合约验证：
   - 用户已注册
   - 购买数量 ≤ maxParticipation
   - 支付足够的 ETH
7. 记录购买信息
```

#### 阶段 5：代币释放（User）

```
1. TGE（Token Generation Event）时刻到达
2. 用户按释放计划调用 withdrawTokens(portionId)
3. 合约检查：
   - 当前时间 ≥ 解锁时间
   - 该部分尚未提取
4. 按比例分配代币给用户
```

## 6. 核心算法详解

### 6.1 质押挖矿奖励算法

这是整个平台最核心的算法，决定了用户如何获得质押奖励。

#### 6.1.1 核心数据结构

```solidity
// 用户信息
struct UserInfo {
    uint256 amount;         // 用户质押的 LP 代币数量
    uint256 rewardDebt;     // 奖励债务
    uint256 tokensUnlockTime; // 质押解锁时间
    address [] salesRegistered; // 注册的销售列表
}

// 池子信息
struct PoolInfo {
    IERC20 lpToken;             // LP 代币合约地址
    uint256 allocPoint;         // 分配点数（权重）
    uint256 lastRewardTimestamp; // 上次奖励计算时间
    uint256 accERC20PerShare;   // 累计每份额奖励（× 1e36）
    uint256 totalDeposits;      // 总质押量
}
```

#### 6.1.2 奖励计算公式

**用户待领取奖励**：

```
pending = (user.amount × pool.accERC20PerShare) ÷ 1e36 - user.rewardDebt
```

**公式解释**：
- `user.amount`：用户当前的质押数量
- `pool.accERC20PerShare`：池子累计每份额奖励（乘以 1e36 提高精度）
- `user.rewardDebt`：用户上次操作时已经计算过的奖励
- 相减得到的是用户自上次操作以来新产生的奖励

#### 6.1.3 accERC20PerShare 更新算法

当池子需要更新奖励时（存款、取款、手动更新触发）：

```
// 1. 计算时间差
nrOfSeconds = min(block.timestamp, endTimestamp) - pool.lastRewardTimestamp

// 2. 计算这段时间内池子应得的奖励
erc20Reward = nrOfSeconds × rewardPerSecond × pool.allocPoint ÷ totalAllocPoint

// 3. 更新累计每份额奖励
pool.accERC20PerShare += erc20Reward × 1e36 ÷ pool.totalDeposits
```

#### 6.1.4 rewardDebt 的作用

`rewardDebt` 是一个巧妙的设计，用于确保：

1. **防止重复计算**：记录用户已经"领取"但未实际提现的奖励
2. **即时结算**：用户每次操作时自动计算并发放待领取奖励
3. **精度控制**：避免浮点数运算，使用整数运算

**更新逻辑**：

```solidity
// 用户操作（存款/取款）后更新 rewardDebt
user.rewardDebt = user.amount × pool.accERC20PerShare ÷ 1e36
```

#### 6.1.5 完整流程示例

```
假设：
- 用户质押 1000 Nova
- 当前 accERC20PerShare = 500 × 10^36
- 用户 rewardDebt = 0

1. 用户质押 1000 Nova：
   user.amount = 1000
   user.rewardDebt = 1000 × 500 × 10^36 ÷ 10^36 = 500,000

2. 过了一段时间，池子产生奖励：
   accERC20PerShare 更新为 600 × 10^36

3. 用户待领取奖励：
   pending = 1000 × 600 × 10^36 ÷ 10^36 - 500,000 = 100,000

4. 用户再质押 500 Nova：
   - 先发放 100,000 奖励
   - user.amount = 1500
   - user.rewardDebt = 1500 × 600 × 10^36 ÷ 10^36 = 900,000
```

### 6.2 线性释放算法 (Vesting)

#### 6.2.1 数据结构

```solidity
struct Vesting {
    uint256[] unlockingTimes;  // 各部分解锁时间点
    uint256[] percents;        // 各部分释放百分比
    uint256 portionVestingPrecision; // 释放精度（如 1000 表示 0.1%）
}
```

#### 6.2.2 释放计算

用户可提取的代币数量：

```
withdrawable = amountBought × percents[portionId] ÷ portionVestingPrecision
```

**示例**：

假设用户购买了 10,000 代币，释放计划如下：

| 部分 | 解锁时间 | 释放百分比 | 可提取数量 |
|------|----------|------------|------------|
| 0 | TGE + 0天 | 20% | 2,000 |
| 1 | TGE + 30天 | 20% | 2,000 |
| 2 | TGE + 60天 | 20% | 2,000 |
| 3 | TGE + 90天 | 20% | 2,000 |
| 4 | TGE + 120天 | 20% | 2,000 |

#### 6.2.3 延迟释放机制

管理员可以延迟所有部分的解锁时间（仅限一次，最多 30 天）：

```solidity
function shiftVestingUnlockingTimes(uint256 timeToShift) external onlyAdmin {
    require(block.timestamp >= sale.tokensUnlockTime, "Too early to shift");
    require(timeToShift <= 2592000, "Max shift is 30 days"); // 30 × 24 × 60 × 60

    for (uint256 i = 0; i < vesting.unlockingTimes.length; i++) {
        vesting.unlockingTimes[i] += timeToShift;
    }
}
```

### 6.3 复利功能 (Compound)

用户可以将待领取的奖励自动复投到质押中：

```solidity
function compound(uint256 _pid) public {
    PoolInfo storage pool = poolInfo[_pid];
    UserInfo storage user = userInfo[_pid][msg.sender];

    // 1. 更新池子
    updatePool(_pid);

    // 2. 计算待领取奖励
    uint256 pendingAmount = user.amount × pool.accERC20PerShare ÷ 1e36 - user.rewardDebt;

    // 3. 将奖励加入质押数量
    user.amount += pendingAmount;
    user.rewardDebt = user.amount × pool.accERC20PerShare ÷ 1e36;

    // 4. 更新池子总质押
    pool.totalDeposits += pendingAmount;
}
```

**优势**：
- 无需先领取再质押，节省 gas 费用
- 自动复投，持续增长质押基数
